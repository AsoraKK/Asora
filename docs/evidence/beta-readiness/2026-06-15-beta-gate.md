## DSR Drill Attempt

- A live `privacy_admin` token was successfully minted against the deployed dev signing configuration using the current Key Vault-backed `JWT_SECRET`.
- Repo and live environment findings on 2026-06-21:
  - The earlier route-shape suspicion was incorrect. The live `GET /api/_admin/dsr/{id}` and `POST /api/_admin/dsr/legal-holds/{id}/clear` paths work when the response wrapper ids are read correctly from `.data.id`.
  - `POST /api/_admin/dsr/legal-holds` succeeds and returns a legal hold id.
  - `POST /api/_admin/dsr/delete` succeeds and returns a queued delete request id.
  - The live Function App now has:
    - `DSR_QUEUE_CONNECTION=DsrQueueStorage`
    - `DsrQueueStorage__queueServiceUri=https://stasoradsrdev.queue.core.windows.net`
  - The Function App managed identity has the required queue roles on the DSR storage account, including `Storage Queue Data Reader` and `Storage Queue Data Message Processor`.
  - The queue trigger function `privacyDsrProcessor` is registered in the live function inventory.
  - The DSR storage queue `dsr-requests` is accumulating delete messages with `dequeueCount: 0`, proving the host is still not consuming them.
- Conclusion: DSR admin auth and route registration are proven, but the legal-hold dry-run is still blocked at live queue consumption. The remaining gap is no longer route mismatch; it is a live Azure Functions queue-trigger execution failure or host/runtime issue.
